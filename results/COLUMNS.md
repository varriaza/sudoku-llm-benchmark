# Benchmark CSV Column Reference

Each row in `benchmark.csv` represents one puzzle run for one model.

---

## Model identity

### `model_name`
The model identifier string returned by the server's `/v1/models` endpoint (or the `name` override in the config).
For llama.cpp this is typically the GGUF filename (e.g. `Qwen3.5-9B-Q4_K_M.gguf`).
For vLLM it is the HuggingFace repo slug (e.g. `meta-llama/Llama-3-8B`).

**Source:** `sudoku_bench/model_info.py` → `detect_model_info()` → `ModelInfo.name`

---

### `model_params`
Parameter count extracted from `model_name` by regex (e.g. `"9B"`, `"70B"`).
Blank when the name contains no recognisable `<digits>B` pattern.

**Source:** `sudoku_bench/model_info.py` → `_extract_params()`

---

### `model_quant`
Quantisation tag extracted from `model_name` by regex (e.g. `"Q4_K_M"`, `"Q8_0"`).
Blank when the name contains no recognisable quant pattern.

**Source:** `sudoku_bench/model_info.py` → `_extract_quant()`

---

## Hardware identity

### `gpu_name`
GPU name string from `nvidia-smi` (e.g. `"NVIDIA GeForce RTX 4080"`).
Blank when nvidia-smi is unavailable or the query fails.

**Source:** `sudoku_bench/gpu_monitor.py` → `GPUMonitor._query_nvidia()` or
`backends/llamacpp/monitor.py` → `LlamaCppMonitor._query_nvidia()`

---

### `gpu_max_vram_mb`
Total installed VRAM in MB from nvidia-smi (the `memory.total` field).
This is a hardware constant, not a measurement. Blank when nvidia-smi is unavailable.

**Source:** same as `gpu_name`

---

## Puzzle identity

### `board_size`
Grid dimensions as a string (e.g. `"4x4"`, `"9x9"`).
Derived from `box_rows * box_cols`.

### `difficulty`
Fraction of cells that are blank in the starting puzzle (0 = fully given, 1 = empty).
E.g. `0.25` means 25 % of cells are blank.

### `puzzle_id`
Unique stable identifier for the puzzle (e.g. `"4x4_d0.25_0001"`).

---

## Solve outcome

### `solved`
`True` if the model submitted a fully correct, fully filled board in any turn.

### `best_pct_correct`
Highest percentage of cells matching the solution across all valid submissions in this run.
`100.0` means the model produced a perfect board at least once.

**Does not track:** intermediate reasoning steps, only the final submitted grids.

### `final_pct_correct`
Percentage of cells matching the solution on the *last* valid submission.
May be lower than `best_pct_correct` if the model regressed.

### `best_num_errors`
Fewest rule violations (duplicates in rows/columns/boxes, modified givens, out-of-range values) in any single submission.
`0` does not imply `solved=True`; the board may have unfilled cells.

### `final_num_errors`
Rule violations in the last valid submission.

---

## Token usage

> **Note:** "valid submissions" does not count malformed responses; those are tracked separately under `malformed_submissions`.

### `num_input_tokens`
`prompt_tokens` from the **last API call only**.
Because the full conversation history is appended on every turn, the last call's `prompt_tokens` is the largest value and the best proxy for context window pressure.

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`

---

### `num_thinking_tokens`
Accumulated `reasoning_tokens` across **all** API calls.
Sourced from `usage.completion_tokens_details.reasoning_tokens`. For non-thinking models this is `0`.
Thinking tokens are part of `completion_tokens` and represent internal chain-of-thought that is not shown to the user, but is re-sent in subsequent prompts (increasing `num_input_tokens` in later turns).

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`

---

### `num_output_tokens`
Accumulated visible output tokens across all API calls: `sum(completion_tokens - reasoning_tokens)`.
Does not include thinking tokens.

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`

---

### `total_response_tokens`
`num_thinking_tokens + num_output_tokens` — all tokens generated by the model.

---

### `total_tokens_used`
`num_input_tokens + total_response_tokens` — rough total token budget consumed.

---

### `context_pct_used`
`num_input_tokens / context_window * 100`, rounded to 2 decimal places.
`100.0` means the run was cut short by the context window guard.
`0.0` is recorded when `context_window` is unknown.

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`, `context_pct` computation

---

## Token throughput

All speed fields are sampled by polling the backend's Prometheus `/metrics` endpoint during the puzzle run (every `gpu_poll_interval` seconds). Blank when the endpoint is unavailable.

**Source (llama.cpp):** derived from counter deltas — `llamacpp:tokens_predicted_total / llamacpp:tokens_predicted_seconds_total` for generation; `llamacpp:prompt_tokens_total / llamacpp:prompt_seconds_total` for prompt. One final sample is taken at `stop()` to capture the completion-time counter update.

**Source (vLLM):** `vllm:avg_generation_throughput_toks_per_s` and `vllm:avg_prompt_throughput_toks_per_s` averaged gauges.

### `avg_gen_toks_per_sec`
Mean generation tokens per second across all samples taken during the run.

### `median_gen_toks_per_sec`
Median generation tokens per second.

### `max_gen_toks_per_sec`
Peak generation tokens per second observed during the run.

### `avg_prompt_toks_per_sec`
Mean prompt-processing tokens per second across all samples.

### `median_prompt_toks_per_sec`
Median prompt-processing tokens per second.

### `max_prompt_toks_per_sec`
Peak prompt-processing tokens per second observed during the run.

---

## Turn and time

### `total_turns`
Number of **valid board submissions** (responses where a parseable grid was found).
Does not count malformed responses.

### `run_started_at`
ISO 8601 timestamp (with timezone offset) recorded just before the first API call.
Example: `2026-03-30T16:41:24-07:00`.

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`, `datetime.now().astimezone()`

---

### `run_finished_at`
ISO 8601 timestamp recorded immediately after the last API response (or context cutoff).

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`, `datetime.now().astimezone()`

---

### `total_seconds`
Wall-clock seconds from first API call to last response (or context cutoff).
Includes network round-trips and server inference time.

---

## VRAM (GPU memory)

All VRAM fields come from polling `nvidia-smi` every `gpu_poll_interval` seconds while the puzzle runs. Blank when nvidia-smi is unavailable.

**Single-GPU only:** queries GPU 0; multi-GPU machines report only the first GPU.

### `avg_vram_mb`
Mean VRAM usage (MB) sampled across the entire puzzle run.

### `max_vram_mb`
Peak VRAM usage (MB) observed during the puzzle run.

---

## Spill detection

### `spilled_to_ram`
Whether inference spilled out of VRAM into system RAM, causing a significant performance drop.

**Detection logic (in priority order):**

1. **KV-cache ratio (llama.cpp only):** `llamacpp:kv_cache_usage_ratio >= 1.0` from the Prometheus `/metrics` endpoint (requires `--metrics` flag on llama-server). Ratio ≥ 1.0 means the KV cache is full and tokens are being offloaded to CPU RAM.

2. **VRAM fallback:** when the KV-cache metric is absent (older llama-server builds or non-llamacpp backends), compares `max_vram_mb > gpu_max_vram_mb`.

3. **`None`:** when neither KV-cache data nor VRAM data is available.

**Source:**
- `backends/llamacpp/monitor.py` → `LlamaCppMonitor._compute_stats()`
- `sudoku_bench/gpu_monitor.py` → `GPUMonitor.stop()`

---

## System RAM

System RAM is always sampled via `psutil.virtual_memory().used` regardless of GPU availability.

**Important:** this is **total system RAM in use by all processes**, not just this benchmark or the model server. It is a proxy for "how much RAM was the machine under while running this puzzle", useful for detecting out-of-VRAM offloading (model layers moved to CPU RAM).

### `avg_sys_ram_mb`
Mean system RAM usage (MB) sampled across the puzzle run.

### `max_sys_ram_mb`
Peak system RAM usage (MB) observed during the puzzle run.

### `total_ram_mb`
`max_vram_mb + max_sys_ram_mb` — a rough upper bound on peak total memory consumed.
Blank when either component is unavailable.

**Source:** `sudoku_bench/runner.py` → `_run_benchmark()`, `total_ram` computation

---

## Submission quality

### `malformed_submissions`
Count of LLM responses in which no parseable sudoku grid was found.
A high value indicates the model frequently ignored the output format instructions.

Does not count turns where a grid was found but contained violations — those are counted in `total_turns` and reflected in `*_num_errors`.

**Source:** `sudoku_bench/runner.py` → `run_puzzle()`, incremented when `parse_board()` returns `None`

---

## Monitor selection

The monitor used depends on the detected backend (logged at run time):

| Backend | Monitor | VRAM source | Spill source |
|---|---|---|---|
| `llamacpp` | `LlamaCppMonitor` | nvidia-smi | KV-cache ratio → VRAM fallback |
| `vllm` | `VLLMMonitor` | `/metrics` Prometheus | Prometheus gauge |
| `unknown` | `GPUMonitor` | nvidia-smi | VRAM fallback only |

Backend detection: `sudoku_bench/model_info.py` → `detect_model_info()`.
llama.cpp is identified by a successful `GET /props` response; vLLM by `/v1/models` alone.
